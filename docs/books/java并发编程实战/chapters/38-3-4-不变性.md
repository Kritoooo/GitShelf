# 3.4 不变性

满⾜同步需求的另⼀种⽅法是使⽤不可变对象（Immutable Object）[EJ Item 13]。到⽬前为⽌，我们介绍了许多与原⼦性和可⻅性相关的问题，例如得到失效数据，丢失更新操作或者观察到某个对象处于不⼀致的状态等等，都与多线程试图同时访问同⼀个可变的状态相关。如果对象的状态不会改变，那么这些问题与复杂性也就⾃然消失了。

如果某个对象在被创建后其状态就不能被修改，那么这个对象就称为不可变对象。线程安全性是不可变对象的固有属性之⼀，它们的不变性条件是由构造函数创建的，只要它们的状态不改变，那么这些不变性条件就能得以维持。

不可变对象⼀定是线程安全的。

不可变对象很简单。它们只有⼀种状态，并且该状态由构造函数来控制。在程序设计中，⼀个最困难的地⽅就是判断复杂对象的可能状态。然⽽，判断不可变对象的状态却很简单。

同样，不可变对象也更加安全。如果将⼀个可变对象传递给不可信的代码，或者将该对象发布到不可信代码可以访问它的地⽅，那么就很危险——不可信代码会改变它们的状态，更糟的是，在代码中将保留⼀个对该对象的引⽤并稍后在其他线程中修改对象的状态。另⼀⽅⾯，不可变对象不会像这样被恶意代码或者有问题的代码破坏，因此可以安全地共享和发布这些对象，⽽⽆须创建保护性的副本[EJ Item24]。

虽然在Java语⾔规范和Java内存模型中都没有给出不可变性的正式定义，但不可变性并不等于将对象中所有的域都声明为final类型，即使

对象中所有的域都是final类型的，这个对象也仍然是可变的，因为在final类型的域中可以保存对可变对象的引⽤。

当满⾜以下条件时，对象才是不可变的：

·对象创建以后其状态就不能修改。

·对象的所有域都是final类型 [1]。

·对象是正确创建的（在对象的创建期间，this引⽤没有逸出）。

在不可变对象的内部仍可以使⽤可变对象来管理它们的状态，如程序清单3-11中的ThreeStooges所⽰。尽管保存姓名的Set对象是可变的，但从ThreeStooges的设计中可以看到，在Set对象构造完成后⽆法对其进⾏修改。stooges是⼀个final类型的引⽤变量，因此所有的对象状态都通过⼀个final域来访问。最后⼀个要求是“正确地构造对象”，这个要求很容易满⾜，因为构造函数能使该引⽤由除了构造函数及其调⽤者之外的代码来访问。
