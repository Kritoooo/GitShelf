不幸的是，UnsafeCountingFactorizer并⾮线程安全的，尽管它在单线程环境中能正确运⾏。与前⾯的UnsafeSequence⼀样，这个类很可能会丢失⼀些更新操作。虽然递增操作 $^ { + + }$ count是⼀种紧凑的语法，使其看上去只是⼀个操作，但这个操作并⾮原⼦的，因⽽它并不会作为⼀个不可分割的操作来执⾏。实际上，它包含了三个独⽴的操作：读取count的值，将值加1，然后将计算结果写⼊count。这是⼀个“读取-修改-写⼊”的操作序列，并且其结果状态依赖于之前的状态。

图1-1给出了两个线程在没有同步的情况下同时对⼀个计数器执⾏递增操作时发⽣的情况。如果计数器的初始值为9，那么在某些情况下，每个线程读到的值都为9，接着执⾏递增操作，并且都将计数器的值设为$1 0 _ { \circ }$ 显然，这并不是我们希望看到的情况，如果有⼀次递增操作丢失了，命中计数器的值就将偏差1。

你可能会认为，在基于Web的服务中，命中计数器值的少量偏差或许是可以接受的，在某些情况下也确实如此。但如果该计数器被⽤来⽣成数值序列或者唯⼀的对象标识符，那么在多次调⽤中返回相同的值将导致严重的数据完整性问题 [1]。在并发编程中，这种由于不恰当的执⾏时序⽽出现不正确的结果是⼀种⾮常重要的情况，它有⼀个正式的名字：竞态条件（Race Condition）。
