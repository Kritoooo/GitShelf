# 2.4 ⽤锁来保护状态

由于锁能使其保护的代码路径以串⾏形式 [1] 来访问，因此可以通过锁来构造⼀些协议以实现对共享状态的独占访问。只要始终遵循这些协议，就能确保状态的⼀致性。

访问共享状态的复合操作，例如命中计数器的递增操作（读取-修改-写⼊）或者延迟初始化（先检查后执⾏），都必须是原⼦操作以避免产⽣竞态条件。如果在复合操作的执⾏过程中持有⼀个锁，那么会使复合操作成为原⼦操作。然⽽，仅仅将复合操作封装到⼀个同步代码块中是不够的。如果⽤同步来协调对某个变量的访问，那么在访问这个变量的所有位置上都需要使⽤同步。⽽且，当使⽤锁来协调对某个变量的访问时，在访问变量的所有位置上都要使⽤同⼀个锁。

⼀种常⻅的错误是认为，只有在写⼊共享变量时才需要使⽤同步，然⽽事实并⾮如此（3.1节将进⼀步解释其中的原因）。

对于可能被多个线程同时访问的可变状态变量，在访问它时都需要持有同⼀个锁，在这种情况下，我们称状态变量是由这个锁保护的。

在程序清单2-6的SynchronizedFactorizer中，lastNumber和lastFactors这两个变量都是由Servlet对象的内置锁来保护的，在标注@GuardedBy中也已经说明了这⼀点。

对象的内置锁与其状态之间没有内在的关联。虽然⼤多数类都将内置锁⽤做⼀种有效的加锁机制，但对象的域并不⼀定要通过内置锁来保护。当获取与对象关联的锁时，并不能阻⽌其他线程访问该对象，某个线程在获得对象的锁之后，只能阻⽌其他线程获得同⼀个锁。之所以每个对象都有⼀个内置锁，只是为了免去显式地创建锁对象。 [2] 你

需要⾃⾏构造加锁协议或者同步策略来实现对共享状态的安全访问，并且在程序中⾃始⾄终地使⽤它们。

每个共享的和可变的变量都应该只由⼀个锁来保护，从⽽使维护⼈员知道是哪⼀个锁。

⼀种常⻅的加锁约定是，将所有的可变状态都封装在对象内部，并通过对象的内置锁对所有访问可变状态的代码路径进⾏同步，使得在该对象上不会发⽣并发访问。在许多线程安全类中都使⽤了这种模式，例如Vector和其他的同步集合类。在这种情况下，对象状态中的所有变量都由对象的内置锁保护起来。然⽽，这种模式并没有任何特殊之处，编译器或运⾏时都不会强制实施这种（或者其他的）模式[3]。 o 如果在添加新的⽅法或代码路径时忘记了使⽤同步，那么这种加锁协议会很容易被破坏。

并⾮所有数据都需要锁的保护，只有被多个线程同时访问的可变数据才需要通过锁来保护。第1章曾介绍，当添加⼀个简单的异步事件时，例如TimerTask，整个程序都需要满⾜线程安全性要求，尤其是当程序状态的封装性⽐较糟糕时。考虑⼀个处理⼤规模数据的单线程程序，由于任何数据都不会在多个线程之间共享，因此在单线程程序中不需要同步。现在，假设希望添加⼀个新功能，即定期地对数据处理进度⽣成快照，这样当程序崩溃或者必须停⽌时⽆须再次从头开始。你可能会选择使⽤TimerTask，每⼗分钟触发⼀次，并将程序状态保存到⼀个⽂件中。

由于TimerTask在另⼀个（由Timer管理的）线程中调⽤，因此现在就有两个线程同时访问快照中的数据：程序的主线程与Timer线程。这意味着，当访问程序的状态时，不仅TimerTask代码必须使⽤同步，⽽且程序中所有访问相同数据的代码路径也必须使⽤同步。原本在程序中不需要使⽤同步，现在变成了在程序的各个位置都需要使⽤同步。

当某个变量由锁来保护时，意味着在每次访问这个变量时都需要⾸先获得锁，这样就确保在同⼀时刻只有⼀个线程可以访问这个变量。当类的不变性条件涉及多个状态变量时，那么还有另外⼀个需求：在不变性条件中的每个变量都必须由同⼀个锁来保护。因此可以在单个原⼦操作中访问或更新这些变量，从⽽确保不变性条件不被破坏。在SynchronizedFactorizer类中说明了这条规则：缓存的数值和因数分解结果都由Servlet对象的内置锁来保护。

对于每个包含多个变量的不变性条件，其中涉及的所有变量都需要由同⼀个锁来保护。

如果同步可以避免竞态条件问题，那么为什么不在每个⽅法声明时都使⽤关键字synchronized？事实上，如果不加区别地滥⽤

synchronized，可能导致程序中出现过多的同步。此外，如果只是将每个⽅法都作为同步⽅法，例如Vector，那么并不⾜以确保Vector上复合操作都是原⼦的：

if（！vector.contains（element））

vector.add（element）；

虽然contains和add等⽅法都是原⼦⽅法，但在上⾯这个“如果不存在则添加（put-if-absent）”的操作中仍然存在竞态条件。虽然synchronized⽅法可以确保单个操作的原⼦性，但如果要把多个操作合并为⼀个复合操作，还是需要额外的加锁机制（请参⻅4.4节了解如何在线程安全对象中添加原⼦操作的⽅法）。此外，将每个⽅法都作为同步⽅法还可能导致活跃性问题（Liveness）或性能问题（Performance），我们在SynchronizedFactorizer中已经看到了这些问题。

[3] 如果某个变量在多个位置上的访问操作中都持有⼀个锁，但并⾮在所有位置上的访问操作都如此时，那么通过⼀些代码核查⼯具，例如FindBugs，就可以发现这种情况，并报告可能出现了⼀个错误。
