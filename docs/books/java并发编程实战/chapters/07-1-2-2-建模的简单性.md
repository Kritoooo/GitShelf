# 1.2.2 建模的简单性

通常，当只需要执⾏⼀种类型的任务（例如修改12个错误）时，在时间管理⽅⾯⽐执⾏多种类型的任务（例如，修复错误、⾯试系统管理员的接任者、完成团队的绩效考核，以及为下个星期的报告做幻灯⽚）要简单。当只有⼀种类型的任务需要完成时，只需埋头⼯作，直到完成所有的任务（或者你已经精疲⼒尽），你不需要花任何精⼒来琢磨下⼀步该做什么。⽽另⼀⽅⾯，如果需要完成多种类型的任务，那么需要管理不同任务之间的优先级和执⾏时间，并在任务之间进⾏切换，这将带来额外的开销。

对于软件来说同样如此：如果在程序中只包含⼀种类型的任务，那么⽐包含多种不同类型任务的程序要更易于编写，错误更少，也更容易测试。如果为模型中每种类型的任务都分配⼀个专门的线程，那么可以形成⼀种串⾏执⾏的假象，并将程序的执⾏逻辑与调度机制的细节，交替执⾏的操作，异步I/O以及资源等待等问题分离开来。通过使⽤线程，可以将复杂并且异步的⼯作流进⼀步分解为⼀组简单并且同步的⼯作流，每个⼯作流在⼀个单独的线程中运⾏，并在特定的同步位置进⾏交互。

我们可以通过⼀些现有的框架来实现上述⽬标，例如Servlet和RMI（Remote Method Invocation，远程⽅法调⽤）。框架负责解决⼀些细节问题，例如请求管理、线程创建、负载平衡，并在正确的时刻将请求分发给正确的应⽤程序组件。编写Servlet的开发⼈员不需要了解有多少请求在同⼀时刻要被处理，也不需要了解套接字的输⼊流或输出流是否被阻塞。当调⽤Servlet的service⽅法来响应Web请求时，可以以同步⽅式来处理这个请求，就好像它是⼀个单线程程序。这种⽅式可以简化组件的开发，并缩短掌握这种框架的学习时间。
