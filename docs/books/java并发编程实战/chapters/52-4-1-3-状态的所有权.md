# 4.1.3 状态的所有权

4.1 节曾指出，如果以某个对象为根节点构造⼀张对象图，那么该对象的状态将是对象图中所有对象包含的域的⼀个⼦集。为什么是⼀个“⼦集”？在从对象可以达到的所有域中，需要满⾜哪些条件才不属于对象状态的⼀部分？

在定义哪些变量将构成对象的状态时，只考虑对象拥有的数据。所有权（Ownership）在Java中并没有得到充分的体现，⽽是属于类设计中的⼀个要素。如果分配并填充了⼀个HashMap对象，那么就相当于创建了多个对象：HashMap对象，在HashMap对象中包含的多个对象，以及在Map.Entry中可能包含的内部对象。HashMap对象的逻辑状态包括所有的Map.Entry对象以及内部对象，即使这些对象都是⼀些独⽴的对象。

⽆论如何，垃圾回收机制使我们避免了如何处理所有权的问题。在$\mathrm { C } { + } { + }$ 中，当把⼀个对象传递给某个⽅法时，必须认真考虑这种操作是否传递对象的所有权，是短期的所有权还是⻓期的所有权。在Java中同样存在这些所有权模型，只不过垃圾回收器为我们减少了许多在引⽤共享⽅⾯常⻅的错误，因此降低了在所有权处理上的开销。

许多情况下，所有权与封装性总是相互关联的：对象封装它拥有的状态，反之也成⽴，即对它封装的状态拥有所有权。状态变量的所有者将决定采⽤何种加锁协议来维持变量状态的完整性。所有权意味着控制权。然⽽，如果发布了某个可变对象的引⽤，那么就不再拥有独占的控制权，最多是“共享控制权”。对于从构造函数或者从⽅法中传递进来的对象，类通常并不拥有这些对象，除⾮这些⽅法是被专门设计为转移传递进来的对象的所有权（例如，同步容器封装器的⼯⼚⽅法）。

容器类通常表现出⼀种“所有权分离”的形式，其中容器类拥有其⾃⾝的状态，⽽客户代码则拥有容器中各个对象的状态。Servlet框架中的ServletContext就是其中⼀个⽰例。ServletContext为Servlet提供了类似于Map形式的对象容器服务，在ServletContext中可以通过名称来注册（setAttribute）或获取（getAttribute）应⽤程序对象。由Servlet容器实现的ServletContext对象必须是线程安全的，因为它肯定会被多个线程同时访问。当调⽤setAttribute和getAttribute时，Servlet不需要使⽤同步，但当使⽤保存在ServletContext中的对象时，则可能需要使⽤同步。这些对象由应⽤程序拥有，Servlet容器只是替应⽤程序保管它们。与所有共享对象⼀样，它们必须安全地被共享。为了防⽌多个线程在并发访问同⼀个对象时产⽣的相互⼲扰，这些对象应该要么是线程安全的对象，要么是事实不可变的对象，或者由锁来保护的对象。[1]

[1] 需要注意的是，虽然HttpSession对象在功能上类似于Servlet框架，但可能有着更严格的要求。由于Servlet容器可能需要访问HttpSession中的对象，以便在复制操作或者钝化操作（Passivation，指的是将状态保存到持久性存储）中对它们序列化，因此这些对象必须是线程安全的，因为容器可能与Web Application程序同时访问它们。（之所以说“可能”，是因为在Servlet的规范中并没有明确定义复制与钝化等操作，这只是⼤多数Servlet容器的⼀个常⻅功能。）
