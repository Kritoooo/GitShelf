# 5.3 阻塞队列和⽣产者-消费者模式

阻塞队列提供了可阻塞的put和take⽅法，以及⽀持定时的offer和poll⽅法。如果队列已经满了，那么put⽅法将阻塞直到有空间可⽤；如果队列为空，那么take⽅法将会阻塞直到有元素可⽤。队列可以是有界的也可以是⽆界的，⽆界队列永远都不会充满，因此⽆界队列上的put⽅法也永远不会阻塞。

阻塞队列⽀持⽣产者-消费者这种设计模式。该模式将“找出需要完成的⼯作”与“执⾏⼯作”这两个过程分离开来，并把⼯作项放⼊⼀个“待完成”列表中以便在随后处理，⽽不是找出后⽴即处理。⽣产者-消费者模式能简化开发过程，因为它消除了⽣产者类和消费者类之间的代码依赖性，此外，该模式还将⽣产数据的过程与使⽤数据的过程解耦开来以简化⼯作负载的管理，因为这两个过程在处理数据的速率上有所不同。

在基于阻塞队列构建的⽣产者-消费者设计中，当数据⽣成时，⽣产者把数据放⼊队列，⽽当消费者准备处理数据时，将从队列中获取数据。⽣产者不需要知道消费者的标识或数量，或者它们是否是唯⼀的⽣产者，⽽只需将数据放⼊队列即可。同样，消费者也不需要知道⽣产者是谁，或者⼯作来⾃何处。BlockingQueue简化了⽣产者-消费者设计的实现过程，它⽀持任意数量的⽣产者和消费者。⼀种最常⻅的⽣产者-消费者设计模式就是线程池与⼯作队列的组合，在Executor任务执⾏框架中就体现了这种模式，这也是第6章和第8章的主题。

以两个⼈洗盘⼦为例，⼆者的劳动分⼯也是⼀种⽣产者-消费者模式：其中⼀个⼈把洗好的盘⼦放在盘架上，⽽另⼀个⼈从盘架上取出盘⼦并把它们烘⼲。在这个⽰例中，盘架相当于阻塞队列。如果盘架上没有盘⼦，那么消费者会⼀直等待，直到有盘⼦需要烘⼲。如果盘架放满了，那么⽣产者会停⽌清洗直到盘架上有更多的空间。我们可以将

这种类⽐扩展为多个⽣产者（虽然可能存在对⽔槽的竞争）和多个消费者，每个⼯⼈只需与盘架打交道。⼈们不需要知道究竟有多少⽣产者或消费者，或者谁⽣产了某个指定的⼯作项。

“⽣产者”和“消费者”的⾓⾊是相对的，某种环境中的消费者在另⼀种不同的环境中可能会成为⽣产者。烘⼲盘⼦的⼯⼈将“消费”洗⼲净的湿盘⼦，⽽产⽣烘⼲的盘⼦。第三个⼈把洗⼲净的盘⼦整理好，在这种情况中，烘⼲盘⼦的⼯⼈既是消费者，也是⽣产者，从⽽就有了两个共享的⼯作队列（每个队列都可能阻塞烘⼲⼯作的运⾏）。

阻塞队列简化了消费者程序的编码，因为take操作会⼀直阻塞直到有可⽤的数据。如果⽣产者不能尽快地产⽣⼯作项使消费者保持忙碌，那么消费者就只能⼀直等待，直到有⼯作可做。在某些情况下，这种⽅式是⾮常合适的（例如，在服务器应⽤程序中，没有任何客户请求服务），⽽在其他⼀些情况下，这也表⽰需要调整⽣产者线程数量和消费者线程数量之间的⽐率，从⽽实现更⾼的资源利⽤率（例如，在“⽹⻚爬⾍[Web Crawler]”或其他应⽤程序中，有⽆穷的⼯作需要完成）。

如果⽣产者⽣成⼯作的速率⽐消费者处理⼯作的速率快，那么⼯作项会在队列中累积起来，最终耗尽内存。同样，put⽅法的阻塞特性也极⼤地简化了⽣产者的编码。如果使⽤有界队列，那么当队列充满时，⽣产者将阻塞并且不能继续⽣成⼯作，⽽消费者就有时间来赶上⼯作处理进度。

阻塞队列同样提供了⼀个offer⽅法，如果数据项不能被添加到队列中，那么将返回⼀个失败状态。这样你就能够创建更多灵活的策略来处理负荷过载的情况，例如减轻负载，将多余的⼯作项序列化并写⼊磁盘，减少⽣产者线程的数量，或者通过某种⽅式来抑制⽣产者线程。

在构建⾼可靠的应⽤程序时，有界队列是⼀种强⼤的资源管理⼯具：它们能抑制并防⽌产⽣过多的⼯作项，使应⽤程序在负荷过载的情况下变得更加健壮。

虽然⽣产者-消费者模式能够将⽣产者和消费者的代码彼此解耦开来，但它们的⾏为仍然会通过共享⼯作队列间接地耦合在⼀起。开发⼈员总会假设消费者处理⼯作的速率能赶上⽣产者⽣成⼯作项的速率，因此通常不会为⼯作队列的⼤⼩设置边界，但这将导致在之后需要重新设计系统架构。因此，应该尽早地通过阻塞队列在设计中构建资源管理机制—这件事情做得越早，就越容易。在许多情况下，阻塞队列能使这项⼯作更加简单，如果阻塞队列并不完全符合设计需求，那么还可以通过信号量（Semaphore）来创建其他的阻塞数据结构（请参⻅5.5.3节）。
