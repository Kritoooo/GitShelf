# 1.4 线程⽆处不在

即使在程序中没有显式地创建线程，但在框架中仍可能会创建线程，因此在这些线程中调⽤的代码同样必须是线程安全的。这将给开发⼈员在设计和实现上带来沉重负担，因为开发线程安全的类⽐开发⾮线程安全的类要更加谨慎和细致。

每个Java应⽤程序都会使⽤线程。当JVM启动时，它将为JVM的内部任务（例如，垃圾收集、终结操作等）创建后台线程，并创建⼀个主线程来运⾏main⽅法。AWT（Abstract Window Toolkit，抽象窗⼝⼯具库）和Swing的⽤户界⾯框架将创建线程来管理⽤户界⾯事件。Timer将创建线程来执⾏延迟任务。⼀些组件框架，例如Servlet和RMI，都会创建线程池并调⽤这些线程中的⽅法。

如果要使⽤这些功能，那么就必须熟悉并发性和线程安全性，因为这些框架将创建线程并且在这些线程中调⽤程序中的代码。虽然将并发性认为是⼀种“可选的”或者“⾼级的”语⾔功能固然理想，但现实情况是，⼏乎所有的Java应⽤程序都是多线程的，因此在使⽤这些框架时仍然需要对应⽤程序状态的访问进⾏协同。

当某个框架在应⽤程序中引⼊并发性时，通常不可能将并发性仅局限于框架代码，因为框架本⾝会回调（Callback）应⽤程序的代码，⽽这些代码将访问应⽤程序的状态。同样，对线程安全性的需求也不能局限于被调⽤的代码，⽽是要延伸到需要访问这些代码所访问的程序状态的所有代码路径。因此，对线程安全性的需求将在程序中蔓延开来。

框架通过在框架线程中调⽤应⽤程序代码将并发性引⼊到程序中。在代码中将不可避免地访问应⽤程序状态，因此所有访问这些状态的代码路径都必须是线程安全的。

下⾯给出的模块都将在应⽤程序之外的线程中调⽤应⽤程序的代码。尽管线程安全性需求可能源⾃这些模块，但却不会⽌步于它们，⽽是会延伸到整个应⽤程序。

Timer。Timer类的作⽤是使任务在稍后的时刻运⾏，或者运⾏⼀次，或者周期性地运⾏。引⼊Timer可能会使串⾏程序变得复杂，因为TimerTask将在Timer管理的线程中执⾏，⽽不是由应⽤程序来管理。如果某个TimerTask访问了应⽤程序中其他线程访问的数据，那么不仅TimerTask需要以线程安全的⽅式来访问数据，其他类也必须采⽤线程安全的⽅式来访问该数据。通常，要实现这个⽬标，最简单的⽅式是确保TimerTask访问的对象本⾝是线程安全的，从⽽就能把线程安全性封装在共享对象内部。

Servlet和JavaServer Page（JSP）。Servlet框架⽤于部署⽹⻚应⽤程序以及分发来⾃HTTP客户端的请求。到达服务器的请求可能会通过⼀个过滤器链被分发到正确的Servlet或JSP。每个Servlet都表⽰⼀个程序逻辑组件，在⾼吞吐率的⽹站中，多个客户端可能同时请求同⼀个Servlet的服务。在Servlet规范中，Servlet同样需要满⾜被多个线程同时调⽤，换句话说，Servlet需要是线程安全的。

即使你可以确保每次只有⼀个线程调⽤某个Servlet，但在构建⽹⻚应⽤程序时仍然必须注意线程安全性。Servlet通常会访问与其他Servlet共享的信息，例如应⽤程序中的对象（这些对象保存在ServletContext中）或者会话中的对象（这些对象保存在每个客户端的HttpSession中）。当⼀个Servlet访问在多个Servlet或者请求中共享的对象时，必须正确地协同对这些对象的访问，因为多个请求可能在不同的线程中同时访问这些对象。Servlet和JSP，以及在ServletContext和HttpSession等容器中保存的Servlet过滤器和对象等，都必须是线程安全的。

远程⽅法调⽤（Remote Method Invocation, RMI）。RMI使代码能够调⽤在其他JVM中运⾏的对象。当通过RMI调⽤某个远程⽅法时，传递

给⽅法的参数必须被打包（也称为列集[Marshaled]）到⼀个字节流中，通过⽹络传输给远程JVM，然后由远程JVM拆包（或者称为散集[Unmarshaled]）并传递给远程⽅法。

当RMI代码调⽤远程对象时，这个调⽤将在哪个线程中执⾏？你并不知道，但肯定不会在你创建的线程中，⽽是将在⼀个由RMI管理的线程中调⽤对象。RMI会创建多少个线程？同⼀个远程对象上的同⼀个远程⽅法会不会在多个RMI线程中被同时调⽤ [1]

远程对象必须注意两个线程安全性问题：正确地协同在多个对象中共享的状态，以及对远程对象本⾝状态的访问（由于同⼀个对象可能会在多个线程中被同时访问）。与Servlet相同，RMI对象应该做好被多个线程同时调⽤的准备，并且必须确保它们⾃⾝的线程安全性。

Swing和AWT。GUI应⽤程序的⼀个固有属性是异步性。⽤户可以在任意时刻选择⼀个菜单项或者按下⼀个按钮，应⽤程序就会及时响应，即使应⽤程序当时正在执⾏其他的任务。Swing和AWT很好地解决了这个问题，它们创建了⼀个单独的线程来处理⽤户触发的事件，并对呈现给⽤户的图形界⾯进⾏更新。

Swing的⼀些组件并不是线程安全的，例如JTable。相反，Swing程序通过将所有对GUI组件的访问局限在事件线程中以实现线程安全性。如果某个应⽤程序希望在事件线程之外控制GUI，那么必须将控制GUI的代码放在事件线程中运⾏。

当⽤户触发某个UI动作时，在事件线程中就会有⼀个事件处理器被调⽤以执⾏⽤户请求的操作。如果事件处理器需要访问由其他线程同时访问的应⽤程序状态（例如编辑某个⽂档），那么这个事件处理器，以及访问这个状态的所有其他代码，都必须采⽤⼀种线程安全的⽅式来访问该状态。
