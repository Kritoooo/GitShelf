# 3.1.4 Volatile变量

Java语⾔提供了⼀种稍弱的同步机制，即volatile变量，⽤来确保将变量的更新操作通知到其他线程。当把变量声明为volatile类型后，编译器与运⾏时都会注意到这个变量是共享的，因此不会将该变量上的操作与其他内存操作⼀起重排序。volatile变量不会被缓存在寄存器或者对其他处理器不可⻅的地⽅，因此在读取volatile类型的变量时总会返回最新写⼊的值。

理解volatile变量的⼀种有效⽅法是，将它们的⾏为想象成程序清单3-3中SynchronizedInteger的类似⾏为，并将volaLile变量的读操作和写操作分别替换为get⽅法和set⽅法 [1]。然⽽，在访问volatile变量时不会执⾏加锁操作，因此也就不会使执⾏线程阻塞，因此volatile变量是⼀种⽐sychronized关键字更轻量级的同步机制。 [2]

volatile变量对可⻅性的影响⽐volatile变量本⾝更为重要。当线程A⾸先写⼊⼀个volatile变量并且线程B随后读取该变量时，在写⼊volatile变量之前对A可⻅的所有变量的值，在B读取了volatile变量后，对B也是可⻅的。因此，从内存可⻅性的⾓度来看，写⼊volatile变量相当于退出同步代码块，⽽读取volatile变量就相当于进⼊同步代码块。然⽽，我们并不建议过度依赖volatile变量提供的可⻅性。如果在代码中依赖volatile变量来控制状态的可⻅性，通常⽐使⽤锁的代码更脆弱，也更难以理解。

仅当volatile变量能简化代码的实现以及对同步策略的验证时，才应该使⽤它们。如果在验证正确性时需要对可⻅性进⾏复杂的判断，那么就不要使⽤volatile变量。volatile变量的正确使⽤⽅式包括：确保它们⾃⾝状态的可⻅性，确保它们所引⽤对象的状态的可⻅性，以及标识⼀些重要的程序⽣命周期事件的发⽣（例如，初始化或关闭）。

程序清单3-4给出了volatile变量的⼀种典型⽤法：检查某个状态标记以判断是否退出循环。在这个⽰例中，线程试图通过类似于数绵⽺的传统⽅法进⼊休眠状态。为了使这个⽰例能正确执⾏，asleep必须为volatile变量。否则，当asleep被另⼀个线程修改时，执⾏判断的线程却发现不了 [3]。我们也可以⽤锁来确保asleep更新操作的可⻅性，但这将使代码变得更加复杂。

虽然volatile变量很⽅便，但也存在⼀些局限性。volatile变量通常⽤做某个操作完成、发⽣中断或者状态的标志，例如程序清单3-4中的asleep标志。尽管volatile变量也可以⽤于表⽰其他的状态信息，但在使⽤时要⾮常⼩⼼。例如，volatile的语义不⾜以确保递增操作（count++）的原⼦性，除⾮你能确保只有⼀个线程对变量执⾏写操作。（原⼦变量提供了“读-改-写”的原⼦操作，并且常常⽤做⼀种“更好的volatile变量”。请参⻅第15章）。

加锁机制既可以确保可⻅性⼜可以确保原⼦性，⽽volatile变量只能确保可⻅性。

当且仅当满⾜以下所有条件时，才应该使⽤volatile变量：

·对变量的写⼊操作不依赖变量的当前值，或者你能确保只有单个线程更新变量的值。

·该变量不会与其他状态变量⼀起纳⼊不变性条件中。

·在访问变量时不需要加锁。
