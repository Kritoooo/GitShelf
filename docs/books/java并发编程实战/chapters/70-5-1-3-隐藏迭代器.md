# 5.1.3 隐藏迭代器

虽然加锁可以防⽌迭代器抛出ConcurrentModificationException，但你必须要记住在所有对共享容器进⾏迭代的地⽅都需要加锁。实际情况要更加复杂，因为在某些情况下，迭代器会隐藏起来，如程序清单5-6中的HiddenIterator所⽰。在HiddenIterator中没有显式的迭代操作，但在粗体标出的代码中将执⾏迭代操作。编译器将字符串的连接操作转换为调⽤StringBuilder.append（Object），⽽这个⽅法⼜会调⽤容器的toString⽅法，标准容器的toString⽅法将迭代容器，并在每个元素上调⽤toString来⽣成容器内容的格式化表⽰。

**程序清单5-6 隐藏在字符串连接中的迭代操作（不要这么做）**

public class HiddenIterator{ @GuardedBy("this") private final Set<Integer>set $\equiv$ newHashSet<Integer> () ; public synchronized void add (Integer i) {set.add (i) ;} public synchronized void remove (Integer i) {set.remove (i) ;} public void addTenThings() { Random r $=$ new Random(); for (int $\mathrm{i = 0}$ . $\mathrm{i <   10}$ . $\mathrm{i + + }$ ) add (r nextInt())； System.out.println("DEBUG:added ten elements to"+set）; }

addTenThings⽅法可能会抛出ConcurrentModificationException，因为在⽣成调试消息的过程中，toString对容器进⾏迭代。当然，真正的问题在于HiddenIteracor不是线程安全的。在使⽤println中的set之前必须⾸先获取HiddenIterator的锁，但在调试代码和⽇志代码中通常会忽视这个要求。

这⾥得到的教训是，如果状态与保护它的同步代码之间相隔越远，那么开发⼈员就越容易忘记在访问状态时使⽤正确的同步。如果HiddenIterator⽤synchronizedSet来包装HashSet，并且对同步代码进⾏封装，那么就不会发⽣这种错误。

正如封装对象的状态有助于维持不变性条件⼀样，封装对象的同步机制同样有助于确保实施同步策略。

容器的hashCode和equals等⽅法也会间接地执⾏迭代操作，当容器作为另⼀个容器的元素或键值时，就会出现这种情况。同样，containsAll、removeAll和retainAll等⽅法，以及把容器作为参数的构造函数，都会对容器进⾏迭代。所有这些间接的迭代操作都可能抛出Concur rentModificationException。
