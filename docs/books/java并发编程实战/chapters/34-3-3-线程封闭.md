# 3.3 线程封闭

当访问共享的可变数据时，通常需要使⽤同步。⼀种避免使⽤同步的⽅式就是不共享数据。如果仅在单线程内访问数据，就不需要同步。这种技术被称为线程封闭（Thread Confinement），它是实现线程安全性的最简单⽅式之⼀。当某个对象封闭在⼀个线程中时，这种⽤法将⾃动实现线程安全性，即使被封闭的对象本⾝不是线程安全的[CPJ2.3.2]。

在Swing中⼤量使⽤了线程封闭技术。Swing的可视化组件和数据模型对象都不是线程安全的，Swing通过将它们封闭到Swing的事件分发线程中来实现线程安全性。要想正确地使⽤Swing，那么在除了事件线程之外的其他线程中就不能访问这些对象（为了进⼀步简化对Swing的使⽤，Swing还提供了invokeLater机制，⽤于将⼀个Runnable实例调度到事件线程中执⾏）。Swing应⽤程序的许多并发错误都是由于错误地在另⼀个线程中使⽤了这些被封闭的对象。

线程封闭技术的另⼀种常⻅应⽤是JDBC（Java DatabaseConnectivity）的Connection对象。JDBC规范并不要求Connection对象必须是线程安全的[1]。在典型的服务器应⽤程序中，线程从连接池中获得⼀个Connection对象，并且⽤该对象来处理请求，使⽤完后再将对象返还给连接池。由于⼤多数请求（例如Servlet请求或EJB调⽤等）都是由单个线程采⽤同步的⽅式来处理，并且在Connection对象返回之前，连接池不会再将它分配给其他线程，因此，这种连接管理模式在处理请求时隐含地将Connection对象封闭在线程中。

在Java语⾔中并没有强制规定某个变量必须由锁来保护，同样在Java语⾔中也⽆法强制将对象封闭在某个线程中。线程封闭是在程序设计中的⼀个考虑因素，必须在程序中实现。Java语⾔及其核⼼库提供了⼀些机制来帮助维持线程封闭性，例如局部变量和ThreadLocal类，但

即便如此，程序员仍然需要负责确保封闭在线程中的对象不会从线程中逸出。
