NumberRange不是线程安全的，没有维持对下界和上界进⾏约束的不变性条件。setLower和setUpper等⽅法都尝试维持不变性条件，但却⽆法做到。setLower和setUpper都是“先检查后执⾏”的操作，但它们没有使⽤⾜够的加锁机制来保证这些操作的原⼦性。假设取值范围为（0，10），如果⼀个线程调⽤setLower（5），⽽另⼀个线程调⽤setUpper（4），那么在⼀些错误的执⾏时序中，这两个调⽤都将通过检查，并且都能设置成功。结果得到的取值范围就是（5，4），那么这是⼀个⽆效的状态。因此，虽然AtomicInteger是线程安全的，但经过组合得到的类却不是。由于状态变量lower和upper不是彼此独⽴的，因此NumberRange不能将线程安全性委托给它的线程安全状态变量。

NumberRange可以通过加锁机制来维护不变性条件以确保其线程安全性，例如使⽤⼀个锁来保护lower和upper。此外，它还必须避免发布lower和upper，从⽽防⽌客户代码破坏其不变性条件。

如果某个类含有复合操作，例如NumberRange，那么仅靠委托并不⾜以实现线程安全性。在这种情况下，这个类必须提供⾃⼰的加锁机制以保证这些复合操作都是原⼦操作，除⾮整个复合操作都可以委托给状态变量。

如果⼀个类是由多个独⽴且线程安全的状态变量组成，并且在所有的操作中都不包含⽆效状态转换，那么可以将线程安全性委托给底层的状态变量。

即使NumberRange的各个状态组成部分都是线程安全的，也不能确保NumberRange的线程安全性，这种问题⾮常类似于3.1.4节介绍的volatile变量规则：仅当⼀个变量参与到包含其他状态变量的不变性条件时，才可以声明为volatile类型。
