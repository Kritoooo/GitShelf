# 2.3.2 重⼊

当某个线程请求⼀个由其他线程持有的锁时，发出请求的线程就会阻塞。然⽽，由于内置锁是可重⼊的，因此如果某个线程试图获得⼀个已经由它⾃⼰持有的锁，那么这个请求就会成功。“重⼊”意味着获取锁的操作的粒度是“线程”，⽽不是“调⽤”[1]。重⼊的⼀种实现⽅法是，为每个锁关联⼀个获取计数值和⼀个所有者线程。当计数值为0时，这个锁就被认为是没有被任何线程持有。当线程请求⼀个未被持有的锁时，JVM将记下锁的持有者，并且将获取计数值置为1。如果同⼀个线程再次获取这个锁，计数值将递增，⽽当线程退出同步代码块时，计数器会相应地递减。当计数值为0时，这个锁将被释放。

重⼊进⼀步提升了加锁⾏为的封装性，因此简化了⾯向对象并发代码的开发。在程序清单2-7的代码中，⼦类改写了⽗类的synchronized⽅法，然后调⽤⽗类中的⽅法，此时如果没有可重⼊的锁，那么这段代码将产⽣死锁。由于Widget和LoggingWidget中doSomething⽅法都是synchronized⽅法，因此每个doSomething⽅法在执⾏前都会获取Widget上的锁。然⽽，如果内置锁不是可重⼊的，那么在调⽤super.doSomething时将⽆法获得Widget上的锁，因为这个锁已经被持有，从⽽线程将永远停顿下去，等待⼀个永远也⽆法获得的锁。重⼊则避免了这种死锁情况的发⽣。
