# 5.4 阻塞⽅法与中断⽅法

线程可能会阻塞或暂停执⾏，原因有多种：等待I/O操作结束，等待获得⼀个锁，等待从Thread.sleep⽅法中醒来，或是等待另⼀个线程的计算结果。当线程阻塞时，它通常被挂起，并处于某种阻塞状态（BLOCKED、WAITING或TIMED_WAITING）。阻塞操作与执⾏时间很⻓的普通操作的差别在于，被阻塞的线程必须等待某个不受它控制的事件发⽣后才能继续执⾏，例如等待I/O操作完成，等待某个锁变成可⽤，或者等待外部计算的结束。当某个外部事件发⽣时，线程被置回RUNNABLE状态，并可以再次被调度执⾏。

BlockingQueue的put和take等⽅法会抛出受检查异常（CheckedException）Interrupted-Exception，这与类库中其他⼀些⽅法的做法相同，例如Thread.sleep。当某⽅法抛出Interrupted-Exception时，表⽰该⽅法是⼀个阻塞⽅法，如果这个⽅法被中断，那么它将努⼒提前结束阻塞状态。

Thread提供了interrupt⽅法，⽤于中断线程或者查询线程是否已经被中断。每个线程都有⼀个布尔类型的属性，表⽰线程的中断状态，当中断线程时将设置这个状态。

中断是⼀种协作机制。⼀个线程不能强制其他线程停⽌正在执⾏的操作⽽去执⾏其他的操作。当线程A中断B时，A仅仅是要求B在执⾏到某个可以暂停的地⽅停⽌正在执⾏的操作—前提是如果线程B愿意停⽌下来。虽然在API或者语⾔规范中并没有为中断定义任何特定应⽤级别的语义，但最常使⽤中断的情况就是取消某个操作。⽅法对中断请求的响应度越⾼，就越容易及时取消那些执⾏时间很⻓的操作。

当在代码中调⽤了⼀个将抛出InterruptedException异常的⽅法时，你⾃⼰的⽅法也就变成了⼀个阻塞⽅法，并且必须要处理对中断的响应。

对于库代码来说，有两种基本选择：

传递InterruptedException。避开这个异常通常是最明智的策略——只需把InterruptedException传递给⽅法的调⽤者。传递InterruptedException的⽅法包括，根本不捕获该异常，或者捕获该异常，然后在执⾏某种简单的清理⼯作后再次抛出这个异常。

恢复中断。有时候不能抛出InterruptedException，例如当代码是Runnable的⼀部分时。在这些情况下，必须捕获InterruptedException，并通过调⽤当前线程上的interrupt⽅法恢复中断状态，这样在调⽤栈中更⾼层的代码将看到引发了⼀个中断，如程序清单5-10所⽰。
