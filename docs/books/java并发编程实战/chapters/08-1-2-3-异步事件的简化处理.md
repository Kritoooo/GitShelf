# 1.2.3 异步事件的简化处理

服务器应⽤程序在接受来⾃多个远程客户端的套接字连接请求时，如果为每个连接都分配其各⾃的线程并且使⽤同步I/O，那么就会降低这类程序的开发难度。

如果某个应⽤程序对套接字执⾏读操作⽽此时还没有数据到来，那么这个读操作将⼀直阻塞，直到有数据到达。在单线程应⽤程序中，这不仅意味着在处理请求的过程中将停顿，⽽且还意味着在这个线程被阻塞期间，对所有请求的处理都将停顿。为了避免这个问题，单线程服务器应⽤程序必须使⽤⾮阻塞I/O，这种I/O的复杂性要远远⾼于同步I/O，并且很容易出错。然⽽，如果每个请求都拥有⾃⼰的处理线程，那么在处理某个请求时发⽣的阻塞将不会影响其他请求的处理。

早期的操作系统通常会将进程中可创建的线程数量限制在⼀个较低的阈值内，⼤约在数百个（甚⾄更少）左右。因此，操作系统提供了⼀些⾼效的⽅法来实现多路I/O，例如Unix的select和poll等系统调⽤，要调⽤这些⽅法，Java类库需要获得⼀组实现⾮阻塞I/O的包（java.nio）。然⽽，在现代操作系统中，线程数量已得到极⼤的提升，这使得在某些平台上，即使有更多的客户端，为每个客户端分配⼀个线程也是可⾏的 [1]。

[1] NPTL线程软件包是专门设计⽤于⽀持数⼗万个线程的，在⼤多数Linux发布版本中都包含了这个软件包。⾮阻塞I/O有其⾃⾝的优势，但如果操作系统能更好地⽀持线程，那么需要使⽤⾮阻塞I/O的情况将变得更少。
