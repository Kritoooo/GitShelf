# 第2章 线程安全性

你或许会感到奇怪，线程或者锁在并发编程中的作⽤，类似于铆钉和⼯字梁在⼟⽊⼯程中的作⽤。要建筑⼀座坚固的桥梁，必须正确地使⽤⼤量的铆钉和⼯字梁。同理，在构建稳健的并发程序时，必须正确地使⽤线程和锁。但这些终归只是⼀些机制。要编写线程安全的代码，其核⼼在于要对状态访问操作进⾏管理，特别是对共享的（Shared）和可变的（Mutable）状态的访问。

从⾮正式的意义上来说，对象的状态是指存储在状态变量（例如实例或静态域）中的数据。对象的状态可能包括其他依赖对象的域。例如，某个HashMap的状态不仅存储在HashMap对象本⾝，还存储在许多Map.Entry对象中。在对象的状态中包含了任何可能影响其外部可⻅⾏为的数据。

“共享”意味着变量可以由多个线程同时访问，⽽“可变”则意味着变量的值在其⽣命周期内可以发⽣变化。我们将像讨论代码那样来讨论线程安全性，但更侧重于如何防⽌在数据上发⽣不受控的并发访问。

⼀个对象是否需要是线程安全的，取决于它是否被多个线程访问。这指的是在程序中访问对象的⽅式，⽽不是对象要实现的功能。要使得对象是线程安全的，需要采⽤同步机制来协同对对象可变状态的访问。如果⽆法实现协同，那么可能会导致数据破坏以及其他不该出现的结果。

当多个线程访问某个状态变量并且其中有⼀个线程执⾏写⼊操作时，必须采⽤同步机制来协同这些线程对变量的访问。Java中的主要同步机制是关键字synchronized，它提供了⼀种独占的加锁⽅式，但“同

步”这个术语还包括volatile类型的变量，显式锁（Explicit Lock）以及原⼦变量。

在上述规则中并不存在⼀些想象中的“例外”情况。即使在某个程序中省略了必要同步机制并且看上去似乎能正确执⾏，⽽且通过了测试并在随后⼏年时间⾥都能正确地执⾏，但程序仍可能在某个时刻发⽣错误。

如果当多个线程访问同⼀个可变的状态变量时没有使⽤合适的同步，那么程序就会出现错误。有三种⽅式可以修复这个问题：

·不在线程之间共享该状态变量。

·将状态变量修改为不可变的变量。

·在访问状态变量时使⽤同步。

如果在设计类的时候没有考虑并发访问的情况，那么在采⽤上述⽅法时可能需要对设计进⾏重⼤修改，因此要修复这个问题可谓是知易⾏难。如果从⼀开始就设计⼀个线程安全的类，那么⽐在以后再将这个类修改为线程安全的类要容易得多。

在⼀些⼤型程序中，要找出多个线程在哪些位置上将访问同⼀个变量是⾮常复杂的。幸运的是，⾯向对象这种技术不仅有助于编写出结构优雅、可维护性⾼的类，还有助于编写出线程安全的类。访问某个变量的代码越少，就越容易确保对变量的所有访问都实现正确同步，同时也更容易找出变量在哪些条件下被访问。Java语⾔并没有强制要求将状态都封装在类中，开发⼈员完全可以将状态保存在某个公开的域（甚⾄公开的静态域）中，或者提供⼀个对内部对象的公开引⽤。然⽽，程序状态的封装性越好，就越容易实现程序的线程安全性，并且代码的维护⼈员也越容易保持这种⽅式。

当设计线程安全的类时，良好的⾯向对象技术、不可修改性，以及明晰的不变性规范都能起到⼀定的帮助作⽤。

在某些情况中，良好的⾯向对象设计技术与实际情况的需求并不⼀致。在这些情况中，可能需要牺牲⼀些良好的设计原则，以换取性能或者对遗留代码的向后兼容。有时候，⾯向对象中的抽象和封装会降低程序的性能（尽管很少有开发⼈员相信），但在编写并发应⽤程序时，⼀种正确的编程⽅法就是：⾸先使代码正确运⾏，然后再提⾼代码的速度。即便如此，最好也只是当性能测试结果和应⽤需求告诉你必须提⾼性能，以及测量结果表明这种优化在实际环境中确实能带来性能提升时，才进⾏优化。 [1]

如果你必须打破封装，那么也并⾮不可以，你仍然可以实现程序的线程安全性，只是更困难，⽽且，程序的线程安全性将更加脆弱，不仅增加了开发的成本和⻛险，⽽且也增加了维护的成本和⻛险。第4章详细介绍了在哪些条件下可以安全地放宽状态变量的封装性。

到⽬前为⽌，我们使⽤了“线程安全类”和“线程安全程序”这两个术语，⼆者的含义基本相同。线程安全的程序是否完全由线程安全类构成？答案是否定的，完全由线程安全类构成的程序并不⼀定就是线程安全的，⽽在线程安全类中也可以包含⾮线程安全的类。第4章还将进⼀步介绍如何对线程安全类进⾏组合的相关问题。在任何情况中，只有当类中仅包含⾃⼰的状态时，线程安全类才是有意义的。线程安全性是⼀个在代码上使⽤的术语，但它只是与状态相关的，因此只能应⽤于封装其状态的整个代码，这可能是⼀个对象，也可能是整个程序。
