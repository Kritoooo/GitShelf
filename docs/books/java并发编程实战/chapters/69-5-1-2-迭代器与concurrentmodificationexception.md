# 5.1.2 迭代器与ConcurrentModificationException

为了将问题阐述清楚，我们使⽤了Vector，虽然这是⼀个“古⽼”的容器类。然⽽，许多“现代”的容器类也并没有消除复合操作中的问题。⽆论在直接迭代还是在Java 5.0引⼊的for-each循环语法中，对容器类进⾏迭代的标准⽅式都是使⽤Iterator。然⽽，如果有其他线程并发地修改容器，那么即使是使⽤迭代器也⽆法避免在迭代期间对容器加锁。在设计同步容器类的迭代器时并没有考虑到并发修改的问题，并且它们表现出的⾏为是“及时失败”（fail-fast）的。这意味着，当它们发现容器在迭代过程中被修改时，就会抛出⼀个

ConcurrentModificationException异常。

这种“及时失败”的迭代器并不是⼀种完备的处理机制，⽽只是“善意地”捕获并发错误，因此只能作为并发问题的预警指⽰器。它们采⽤的实现⽅式是，将计数器的变化与容器关联起来：如果在迭代期间计数器被修改，那么hasNext或next将抛出

ConcurrentModificationException。然⽽，这种检查是在没有同步的情况下进⾏的，因此可能会看到失效的计数值，⽽迭代器可能并没有意识到已经发⽣了修改。这是⼀种设计上的权衡，从⽽降低并发修改操作的检测代码 [1] 对程序性能带来的影响。

程序清单5-5说明了如何使⽤for-each循环语法对List容器进⾏迭代。从内部来看，javac将⽣成使⽤Iterator的代码，反复调⽤hasNext和next来迭代List对象。与迭代Vector⼀样，要想避免出现

ConcurrentModificationException，就必须在迭代过程持有容器的锁。

然⽽，有时候开发⼈员并不希望在迭代期间对容器加锁。例如，某些线程在可以访问容器之前，必须等待迭代过程结束，如果容器的规模很⼤，或者在每个元素上执⾏操作的时间很⻓，那么这些线程将⻓时间等待。同样，如果容器像程序清单5-4中那样加锁，那么在调⽤doSomething时将持有⼀个锁，这可能会产⽣死锁（请参⻅第10章）。即使不存在饥饿或者死锁等⻛险，⻓时间地对容器加锁也会降低程序的可伸缩性。持有锁的时间越⻓，那么在锁上的竞争就可能越激烈，如果许多线程都在等待锁被释放，那么将极⼤地降低吞吐量和CPU的利⽤率（请参⻅第11章）。

如果不希望在迭代期间对容器加锁，那么⼀种替代⽅法就是“克隆”容器，并在副本上进⾏迭代。由于副本被封闭在线程内，因此其他线程不会在选代期间对其进⾏修改，这样就避免了抛出

ConcurrentModificationException（在克隆过程中仍然需要对容器加锁）。在克隆容器时存在显著的性能开销。这种⽅式的好坏取决于多个因素，包括容器的⼤⼩，在每个元素上执⾏的⼯作，迭代操作相对于容器其他操作的调⽤频率，以及在响应时间和吞吐量等⽅⾯的需求。

[1] 在单线程代码中也可能抛出ConcurrentModificationException异常。当对象直接从容器中删除⽽不是通过Iterator.remove来删除时，就会抛出这个异常。
