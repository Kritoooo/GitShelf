# 1.1 并发简史

在早期的计算机中不包含操作系统，它们从头到尾只执⾏⼀个程序，并且这个程序能访问计算机中的所有资源。在这种裸机环境中，不仅很难编写和运⾏程序，⽽且每次只能运⾏⼀个程序，这对于昂贵并且稀有的计算机资源来说也是⼀种浪费。

操作系统的出现使得计算机每次能运⾏多个程序，并且不同的程序都在单独的进程中运⾏：操作系统为各个独⽴执⾏的进程分配各种资源，包括内存，⽂件句柄以及安全证书等。如果需要的话，在不同的进程之间可以通过⼀些粗粒度的通信机制来交换数据，包括：套接字、信号处理器、共享内存、信号量以及⽂件等。

之所以在计算机中加⼊操作系统来实现多个程序的同时执⾏，主要是基于以下原因：

资源利⽤率。在某些情况下，程序必须等待某个外部操作执⾏完成，例如输⼊操作或输出操作等，⽽在等待时程序⽆法执⾏其他任何⼯作。因此，如果在等待的同时可以运⾏另⼀个程序，那么⽆疑将提⾼资源的利⽤率。

公平性。不同的⽤户和程序对于计算机上的资源有着同等的使⽤权。⼀种⾼效的运⾏⽅式是通过粗粒度的时间分⽚（Time Slicing）使这些⽤户和程序能共享计算机资源，⽽不是由⼀个程序从头运⾏到尾，然后再启动下⼀个程序。

便利性。通常来说，在计算多个任务时，应该编写多个程序，每个程序执⾏⼀个任务并在必要时相互通信，这⽐只编写⼀个程序来计算所有任务更容易实现。

在早期的分时系统中，每个进程相当于⼀台虚拟的冯·诺依曼计算机，它拥有存储指令和数据的内存空间，根据机器语⾔的语义以串⾏⽅式执⾏指令，并通过⼀组I/O指令与外部设备通信。对每条被执⾏的指令，都有相应的“下⼀条指令”，程序中的控制流是按照指令集的规则来确定的。当前，⼏乎所有的主流编程语⾔都遵循这种串⾏编程模型，并且在这些语⾔的规范中也都清晰地定义了在某个动作完成之后需要执⾏的“下⼀个动作”。

串⾏编程模型的优势在于其直观性和简单性，因为它模仿了⼈类的⼯作⽅式：每次只做⼀件事情，做完之后再做另⼀件。例如，⾸先起床，穿上睡⾐，然后下楼，喝早茶。在编程语⾔中，这些现实世界中的动作可以被进⼀步抽象为⼀组粒度更细的动作。例如，喝早茶的动作可以被进⼀步细化为：打开橱柜，挑选喜欢的茶叶，将⼀些茶叶倒⼊杯中，看看茶壶中是否有⾜够的⽔，如果没有的话加些⽔，将茶壶放到⽕炉上，点燃⽕炉，然后等⽔烧开等等。在最后⼀步等⽔烧开的过程中包含了⼀定程度的异步性。当正在烧⽔时，你可以⼲等着，也可以做些其他事情，例如开始烤⾯包（这是另⼀个异步任务）或者看报纸，同时留意茶壶⽔是否烧开。茶壶和⾯包机的⽣产商都很清楚：⽤户通常会采⽤异步⽅式来使⽤他们的产品，因此当这些机器完成任务时都会发出声⾳提⽰。但凡做事⾼效的⼈，总能在串⾏性与异步性之间找到合理的平衡，对于程序来说同样如此。

这些促使进程出现的因素（资源利⽤率、公平性以及便利性等）同样也促使着线程的出现。线程允许在同⼀个进程中同时存在多个程序控制流。线程会共享进程范围内的资源，例如内存句柄和⽂件句柄，但每个线程都有各⾃的程序计数器（Program Counter）、栈以及局部变量等。线程还提供了⼀种直观的分解模式来充分利⽤多处理器系统中的硬件并⾏性，⽽在同⼀个程序中的多个线程也可以被同时调度到多个CPU上运⾏。

线程也被称为轻量级进程。在⼤多数现代操作系统中，都是以线程为基本的调度单位，⽽不是进程。如果没有明确的协同机制，那么线程将彼此独⽴执⾏。由于同⼀个进程中的所有线程都将共享进程的内存地址空间，因此这些线程都能访问相同的变量并在同⼀个堆上分配对象，这就需要实现⼀种⽐在进程间共享数据粒度更细的数据共享机制。如果没有明确的同步机制来协同对共享数据的访问，那么当⼀个线程正在使⽤某个变量时，另⼀个线程可能同时访问这个变量，这将造成不可预测的结果。
