# 2.2.1 竞态条件

在UnsafeCountingFactorizer中存在多个竞态条件，从⽽使结果变得不可靠。当某个计算的正确性取决于多个线程的交替执⾏时序时，那么就会发⽣竞态条件。换句话说，就是正确的结果要取决于运⽓ [2]。最常⻅的竞态条件类型就是“先检查后执⾏（Check-Then-Act）”操作，即通过⼀个可能失效的观测结果来决定下⼀步的动作。

在实际情况中经常会遇到竞态条件。例如，假定你计划中午在University Avenue的星巴克与⼀位朋友会⾯。但当你到达那⾥时，发现

在University Avenue上有两家星巴克，并且你不知道说好碰⾯的是哪⼀家。在12:10时，你没有在星巴克A看到朋友，那么就会去星巴克B看看他是否在那⾥，但他也不在那⾥。这有⼏种可能：你的朋友迟到了，还没到任何⼀家星巴克；你的朋友在你离开后到了星巴克A；你的朋友在星巴克B，但他去星巴克A找你，并且此时正在去星巴克A的途中。我们假设是最糟糕的情况，即最后⼀种可能。现在是12:15，你们两个都去过了两家星巴克，并且都开始怀疑对⽅是否失约了。现在你会怎么做？回到另⼀家星巴克？来来回回要⾛多少次？除⾮你们之间约定了某种协议，否则你们整天都在University Avenue上⾛来⾛去，倍感沮丧。

在“我去看看他是否在另⼀家星巴克”这种⽅法中，问题在于：当你在街上⾛时，你的朋友可能已经离开了你要去的星巴克。你⾸先看了看星巴克A，发现“他不在”，并且开始去找他。你可以在星巴克B中做同样的选择，但不是同时发⽣。两家星巴克之间有⼏分钟的路程，⽽就在这⼏分钟的时间⾥，系统的状态可能会发⽣变化。

在星巴克这个⽰例中说明了⼀种竞态条件，因为要获得正确的结果（与朋友会⾯），必须取决于事件的发⽣时序（当你们到达星巴克时，在离开并去另⼀家星巴克之前会等待多⻓时间……）。当你迈出前门时，你在星巴克A的观察结果将变得⽆效，你的朋友可能从后门进来了，⽽你却不知道。这种观察结果的失效就是⼤多数竞态条件的本质—基于⼀种可能失效的观察结果来做出判断或者执⾏某个计算。这种类型的竞态条件称为“先检查后执⾏”：⾸先观察到某个条件为真（例如⽂件X不存在），然后根据这个观察结果采⽤相应的动作（创建⽂件X），但事实上，在你观察到这个结果以及开始创建⽂件之间，观察结果可能变得⽆效（另⼀个线程在这期间创建了⽂件X），从⽽导致各种问题（未预期的异常、数据被覆盖、⽂件被破坏等）。

[1] 在UnsafeSequence和UnsafeCountingFactorizer中还存在其他⼀些严重的问题，例如可能出现失效数据（Stale Data）问题（3.1.1节）。  
[2] 竞态条件这个术语很容易与另⼀个相关术语“数据竞争（DataRace）”相混淆。数据竞争是指，如果在访问共享的⾮final类型的域时没有采⽤同步来进⾏协同，那么就会出现数据竞争。当⼀个线程写⼊⼀个变量⽽另⼀个线程接下来读取这个变量，或者读取⼀个之前由另⼀个线程写⼊的变量时，并且在这两个线程之间没有使⽤同步，那么就可能出现数据竞争。在Java内存模型中，如果在代码中存在数据竞争，那么这段代码就没有确定的语义。并⾮所有的竞态条件都是数据竞争，同样并⾮所有的数据竞争都是竞态条件，但⼆者都可能使并发程序失败。在UnsafeCountingFactorizer中既存在竞态条件，⼜存在数据竞争。参⻅第16章了解数据竞争的更详细内容。
