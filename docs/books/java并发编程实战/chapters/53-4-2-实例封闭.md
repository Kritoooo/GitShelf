# 4.2 实例封闭

如果某对象不是线程安全的，那么可以通过多种技术使其在多线程程序中安全地使⽤。你可以确保该对象只能由单个线程访问（线程封闭），或者通过⼀个锁来保护对该对象的所有访问。

封装简化了线程安全类的实现过程，它提供了⼀种实例封闭机制（Instance Confinement），通常也简称为“封闭”[CPJ 2.3.3]。当⼀个对象被封装到另⼀个对象中时，能够访问被封装对象的所有代码路径都是已知的。与对象可以由整个程序访问的情况相⽐，更易于对代码进⾏分析。通过将封闭机制与合适的加锁策略结合起来，可以确保以线程安全的⽅式来使⽤⾮线程安全的对象。

将数据封装在对象内部，可以将数据的访问限制在对象的⽅法上，从⽽更容易确保线程在访问数据时总能持有正确的锁。

被封闭对象⼀定不能超出它们既定的作⽤域。对象可以封闭在类的⼀个实例（例如作为类的⼀个私有成员）中，或者封闭在某个作⽤域内（例如作为⼀个局部变量），再或者封闭在线程内（例如在某个线程中将对象从⼀个⽅法传递到另⼀个⽅法，⽽不是在多个线程之间共享该对象）。当然，对象本⾝不会逸出—出现逸出情况的原因通常是由于开发⼈员在发布对象时超出了对象既定的作⽤域。

程序清单4-2中的PersonSet说明了如何通过封闭与加锁等机制使⼀个类成为线程安全的（即使这个类的状态变量并不是线程安全的）。PersonSet的状态由HashSet来管理的，⽽HashSet并⾮线程安全的。但由于mySet是私有的并且不会逸出，因此HashSet被封闭在PersonSet中。唯⼀能访问mySet的代码路径是addPerson与containsPerson，在执⾏它们时都要获得PersonSet上的锁。PersonSet的状态完全由它的内置锁保护，因⽽PersonSet是⼀个线程安全的类。

在Java平台的类库中还有很多线程封闭的⽰例，其中有些类的唯⼀⽤途就是将⾮线程安全的类转化为线程安全的类。⼀些基本的容器类并⾮线程安全的，例如ArrayList和HashMap，但类库提供了包装器⼯⼚⽅法（例如Collections.synchronizedList及其类似⽅法），使得这些⾮线程安全的类可以在多线程环境中安全地使⽤。这些⼯⼚⽅法通过“装饰器（Decorator）”模式（Gamma et al.，1995）将容器类封装在⼀个同步的包装器对象中，⽽包装器能将接⼝中的每个⽅法都实现为同步⽅法，并将调⽤请求转发到底层的容器对象上。只要包装器对象拥有对底层容器对象的唯⼀引⽤（即把底层容器对象封闭在包装器中），那么它就是线程安全的。在这些⽅法的Javadoc中指出，对底层容器对象的所有访问必须通过包装器来进⾏。

当然，如果将⼀个本该被封闭的对象发布出去，那么也能破坏封闭性。如果⼀个对象本应该封闭在特定的作⽤域内，那么让该对象逸出作⽤域就是⼀个错误。当发布其他对象时，例如迭代器或内部的类实例，可能会间接地发布被封闭对象，同样会使被封闭对象逸出。

封闭机制更易于构造线程安全的类，因为当封闭类的状态时，在分析类的线程安全性时就⽆须检查整个程序。
