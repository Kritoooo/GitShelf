# 1.2.4 响应更灵敏的⽤户界⾯

传统的GUI应⽤程序通常都是单线程的，从⽽在代码的各个位置都需要调⽤poll⽅法来获得输⼊事件（这种⽅式将给代码带来极⼤的混乱），或者通过⼀个“主事件循环（Main Event Loop）”来间接地执⾏应⽤程序的所有代码。如果在主事件循环中调⽤的代码需要很⻓时间才能执⾏完成，那么⽤户界⾯就会“冻结”，直到代码执⾏完成。这是因为只有当执⾏控制权返回到主事件循环后，才能处理后续的⽤户界⾯事件。

在现代的GUI框架中，例如AWT和Swing等⼯具，都采⽤⼀个事件分发线程（Event Dispatch Thread, EDT）来替代主事件循环。当某个⽤户界⾯事件发⽣时（例如按下⼀个按钮），在事件线程中将调⽤应⽤程序的事件处理器。由于⼤多数GUI框架都是单线程⼦系统，因此到⽬前为⽌仍然存在主事件循环，但它现在处于GUI⼯具的控制下并在其⾃⼰的线程中运⾏，⽽不是在应⽤程序的控制下。

如果在事件线程中执⾏的任务都是短暂的，那么界⾯的响应灵敏度就较⾼，因为事件线程能够很快地处理⽤户的动作。然⽽，如果事件线程中的任务需要很⻓的执⾏时间，例如对⼀个⼤型⽂档进⾏拼写检查，或者从⽹络上获取⼀个资源，那界⾯的响应灵敏度就会降低。如果⽤户在执⾏这类任务时触发了某个动作，那么必须等待很⻓时间才能获得响应，因为事件线程要先执⾏完该任务。更糟糕的是，不仅界⾯失去响应，⽽且即使在界⾯上包含了“取消”按钮，也⽆法取消这个⻓时间执⾏的任务，因为事件线程只有在执⾏完该任务后才能响应“取消”按钮的点击事件。然⽽，如果将这个⻓时间运⾏的任务放在⼀个单独的线程中运⾏，那么事件线程就能及时地处理界⾯事件，从⽽使⽤户界⾯具有更⾼的灵敏度。
