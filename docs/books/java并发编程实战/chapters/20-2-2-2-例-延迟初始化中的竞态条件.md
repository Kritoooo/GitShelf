使⽤“先检查后执⾏”的⼀种常⻅情况就是延迟初始化。延迟初始化的⽬的是将对象的初始化操作推迟到实际被使⽤时才进⾏，同时要确保只被初始化⼀次。在程序清单2-3中的LazyInitRace说明了这种延迟初始化情况。getInstance⽅法⾸先判断ExpensiveObject是否已经被初始化，如果已经初始化则返回现有的实例，否则，它将创建⼀个新的实例，并返回⼀个引⽤，从⽽在后来的调⽤中就⽆须再执⾏这段⾼开销的代码路径。

空。此时的instance是否为空，要取决于不可预测的时序，包括线程的调度⽅式，以及A需要花多⻓时间来初始化ExpensiveObject并设置instance。如果当B检查时，instance为空，那么在两次调⽤getInstance时可能会得到不同的结果，即使getInstance通常被认为是返回相同的实例。

在UnsafeCountingFactorizer的统计命中计数操作中存在另⼀种竞态条件。在“读取-修改-写⼊”这种操作（例如递增⼀个计数器）中，基于对象之前的状态来定义对象状态的转换。要递增⼀个计数器，你必须知道它之前的值，并确保在执⾏更新的过程中没有其他线程会修改或使⽤这个值。

与⼤多数并发错误⼀样，竞态条件并不总是会产⽣错误，还需要某种不恰当的执⾏时序。然⽽，竞态条件也可能导致严重的问题。假定LazyInitRace被⽤于初始化应⽤程序范围内的注册表，如果在多次调⽤中返回不同的实例，那么要么会丢失部分注册信息，要么多个⾏为对同⼀组注册对象表现出不⼀致的视图。如果将UnsafeSequence⽤于在某个持久化框架中⽣成对象的标识，那么两个不同的对象最终将获得相同的标识，这就违反了标识的完整性约束条件。
