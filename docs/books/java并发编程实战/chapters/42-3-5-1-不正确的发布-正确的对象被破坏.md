# 3.5.1 不正确的发布：正确的对象被破坏

你不能指望⼀个尚未被完全创建的对象拥有完整性。某个观察该对象的线程将看到对象处于不⼀致的状态，然后看到对象的状态突然发⽣变化，即使线程在对象发布后还没有修改过它。事实上，如果程序清

单3-15中的Holder使⽤程序清单3-14中的不安全发布⽅式，那么另⼀个线程在调⽤assertSanity时将抛出AssertionError。 [1]

程序清单3-15 由于未被正确发布，因此这个类可能出现故障  
public class Holder{ private int n; public Holder (int n) {this.n=n;} public void assertSanity() { if $(n! = n)$ throw new AssertionError("This statement is false."); }

由于没有使⽤同步来确保Holder对象对其他线程可⻅，因此将Holder称为“未被正确发布”。在未被正确发布的对象中存在两个问题。⾸先，除了发布对象的线程外，其他线程可以看到的Holder域是⼀个失效值，因此将看到⼀个空引⽤或者之前的旧值。然⽽，更糟糕的情况是，线程看到Holder引⽤的值是最新的，但Holder状态的值却是失效的 [2]。情况变得更加不可预测的是，某个线程在第⼀次读取域时得到失效值，⽽再次读取这个域时会得到⼀个更新值，这也是assertSainty抛出AssertionError的原因。

如果没有⾜够的同步，那么当在多个线程间共享数据时将发⽣⼀些⾮常奇怪的事情。

[1] 问题并不在于Holder类本⾝，⽽是在于Holder类未被正确地发布。然⽽，如果将n声明为final类型，那么Holder将不可变，从⽽避免出现不正确发布的问题。请参⻅3.5.2节。  
[2] 尽管在构造函数中设置的域值似乎是第⼀次向这些域中写⼊的值，因此不会有“更旧的”值被视为失效值，但Object的构造函数会在⼦类构造函数运⾏之前先将默认值写⼊所有的域。因此，某个域的默认值可能被视为失效值。
