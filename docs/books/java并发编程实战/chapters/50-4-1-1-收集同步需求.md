# 4.1.1 收集同步需求

要确保类的线程安全性，就需要确保它的不变性条件不会在并发访问的情况下被破坏，这就需要对其状态进⾏推断。对象与变量都有⼀个状态空间，即所有可能的取值。状态空间越⼩，就越容易判断线程的状态。final类型的域使⽤得越多，就越能简化对象可能状态的分析过程。（在极端的情况中，不可变对象只有唯⼀的状态。）

在许多类中都定义了⼀些不可变条件，⽤于判断状态是有效的还是⽆效的。Counter中的value域是long类型的变量，其状态空间为从Long.MIN_VALUE到Long.MAX_VALUE，但Counter中value在取值范围上存在着⼀个限制，即不能是负值。

同样，在操作中还会包含⼀些后验条件来判断状态迁移是否是有效的。如果Counter的当前状态为17，那么下⼀个有效状态只能是 $1 8 _ { \circ }$ 当下⼀个状态需要依赖当前状态时，这个操作就必须是⼀个复合操作。并⾮所有的操作都会在状态转换上施加限制，例如，当更新⼀个保存当前温度的变量时，该变量之前的状态并不会影响计算结果。

由于不变性条件以及后验条件在状态及状态转换上施加了各种约束，因此就需要额外的同步与封装。如果某些状态是⽆效的，那么必须对底层的状态变量进⾏封装，否则客户代码可能会使对象处于⽆效状态。如果在某个操作中存在⽆效的状态转换，那么该操作必须是原⼦的。另外，如果在类中没有施加这种约束，那么就可以放宽封装性或序列化等需求，以便获得更⾼的灵活性或性能。

在类中也可以包含同时约束多个状态变量的不变性条件。在⼀个表⽰数值范围的类（例如程序清单4-10中的NumberRange）中可以包含两个状态变量，分别表⽰范围的上界和下界。这些变量必须遵循的约束是，下界值应该⼩于或等于上界值。类似于这种包含多个变量的不变性条件将带来原⼦性需求：这些相关的变量必须在单个原⼦操作中进

⾏读取或更新。不能⾸先更新⼀个变量，然后释放锁并再次获得锁，然后再更新其他的变量。因为释放锁后，可能会使对象处于⽆效状态。如果在⼀个不变性条件中包含多个变量，那么在执⾏任何访问相关变量的操作时，都必须持有保护这些变量的锁。

如果不了解对象的不变性条件与后验条件，那么就不能确保线程安全性。要满⾜在状态变量的有效值或状态转换上的各种约束条件，就需要借助于原⼦性与封装性。
