# 2.1 什么是线程安全性

要对线程安全性给出⼀个确切的定义是⾮常复杂的。定义越正式，就越复杂，不仅很难提供有实际意义的指导建议，⽽且也很难从直观上

去理解。因此，下⾯给出了⼀些⾮正式的描述，看上去令⼈困惑。在互联⽹上可以搜索到许多“定义”，例如：

……可以在多个线程中调⽤，并且在线程之间不会出现错误的交互。

……可以同时被多个线程调⽤，⽽调⽤者⽆须执⾏额外的动作。

看看这些定义，难怪我们会对线程安全性感到困惑。它们听起来⾮常像“如果某个类可以在多个线程中安全地使⽤，那么它就是⼀个线程安全的类”。对于这种说法，虽然没有太多的争议，但同样也不会带来太多的帮助。我们如何区分线程安全的类以及⾮线程安全的类？进⼀步说，“安全”的含义是什么？

在线程安全性的定义中，最核⼼的概念就是正确性。如果对线程安全性的定义是模糊的，那么就是因为缺乏对正确性的清晰定义。

正确性的含义是，某个类的⾏为与其规范完全⼀致。在良好的规范中通常会定义各种不变性条件（Invariant）来约束对象的状态，以及定义各种后验条件（Postcondition）来描述对象操作的结果。由于我们通常不会为类编写详细的规范，那么如何知道这些类是否正确呢？我们⽆法知道，但这并不妨碍我们在确信“类的代码能⼯作”后使⽤它们。这种“代码可信性”⾮常接近于我们对正确性的理解，因此我们可以将单线程的正确性近似定义为“所⻅即所知（we know it when we seeit）”。在对“正确性”给出了⼀个较为清晰的定义后，就可以定义线程安全性：当多个线程访问某个类时，这个类始终都能表现出正确的⾏为，那么就称这个类是线程安全的。

当多个线程访问某个类时，不管运⾏时环境采⽤何种调度⽅式或者这些线程将如何交替执⾏，并且在主调代码中不需要任何额外的同步或协同，这个类都能表现出正确的⾏为，那么就称这个类是线程安全的。

由于单线程程序也可以看成是⼀个多线程程序，如果某个类在单线程环境[2] 中都不是正确的，那么它肯定不会是线程安全的。如果正确地实现了某个对象，那么在任何操作中（包括调⽤对象的公有⽅法或者对其公有域进⾏读/写操作）都不会违背不变性条件或后验条件。在线程安全类的对象实例上执⾏的任何串⾏或并⾏操作都不会使对象处于⽆效状态。

在线程安全类中封装了必要的同步机制，因此客户端⽆须进⼀步采取同步措施。

⽰例：⼀个⽆状态的Servlet

我们在第1章列出了⼀组框架，其中每个框架都能创建多个线程并在这些线程中调⽤你编写的代码，因此你需要保证编写的代码是线程安全的。通常，线程安全性的需求并⾮来源于对线程的直接使⽤，⽽是使⽤像Servlet这样的框架。我们来看⼀个简单的⽰例—⼀个基于Servlet的因数分解服务，并逐渐扩展它的功能，同时确保它的线程安全性。

程序清单2-1给出了⼀个简单的因数分解Servlet。这个Servlet从请求中提取出数值，执⾏因数分解，然后将结果封装到该Servlet的响应中。

**程序清单2-1 ⼀个⽆状态的Servlet**

@ThreadSafe

public class StatelessFactorizer implements Servlet{

public void service（ServletRequest req, ServletResponse resp）{

BigInteger i=extractFromRequest（req）；

BigInteger[]factors=factor（i）；

encodeIntoResponse（resp, factors）；  
}   
}

与⼤多数Servlet相同，StatelessFactorizer是⽆状态的：它既不包含任何域，也不包含任何对其他类中域的引⽤。计算过程中的临时状态仅存在于线程栈上的局部变量中，并且只能由正在执⾏的线程访问。访问StatelessFactorizer的线程不会影响另⼀个访问同⼀个StatelessFactorizer的线程的计算结果，因为这两个线程并没有共享状态，就好像它们都在访问不同的实例。由于线程访问⽆状态对象的⾏为并不会影响其他线程中操作的正确性，因此⽆状态对象是线程安全的。

⽆状态对象⼀定是线程安全的。

⼤多数Servlet都是⽆状态的，从⽽极⼤地降低了在实现Servlet线程安全性时的复杂性。只有当Servlet在处理请求时需要保存⼀些信息，线程安全性才会成为⼀个问题。

[1]在编写并发代码时，应该始终遵循这个原则。由于并发错误是⾮常难以重现和调试的，因此如果只是在某段很少执⾏的代码路径上获得了性能提升，那么很可能被程序运⾏时存在的失败⻛险⽽抵消。  
[2] 如果你觉得这⾥对“正确性”的定义有些模糊，那么可以将线程安全类认为是⼀个在并发环境和单线程环境中都不会被破坏的类。
