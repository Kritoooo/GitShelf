# 5.2.1 ConcurrentHashMap

同步容器类在执⾏每个操作期间都持有⼀个锁。在⼀些操作中，例如HashMap.get或List.contains，可能包含⼤量的⼯作：当遍历散列桶或链表来查找某个特定的对象时，必须在许多元素上调⽤equals（⽽equals本⾝还包含⼀定的计算量）。在基于散列的容器中，如果hashCode不能很均匀地分布散列值，那么容器中的元素就不会均匀地分布在整个容器中。某些情况下，某个糟糕的散列函数还会把⼀个散列表变成线性链表。当遍历很⻓的链表并且在某些或者全部元素上调⽤equals⽅法时，会花费很⻓的时间，⽽其他线程在这段时间内都不能访问该容器。

与HashMap⼀样，ConcurrentHashMap也是⼀个基于散列的Map，但它使⽤了⼀种完全不同的加锁策略来提供更⾼的并发性和伸缩性。

ConcurrentHashMap并不是将每个⽅法都在同⼀个锁上同步并使得每次只能有⼀个线程访问容器，⽽是使⽤⼀种粒度更细的加锁机制来实现更⼤程度的共享，这种机制称为分段锁（Lock Striping，请参⻅11.4.3节）。在这种机制中，任意数量的读取线程可以并发地访问Map，执⾏读取操作的线程和执⾏写⼊操作的线程可以并发地访问Map，并且⼀定数量的写⼊线程可以并发地修改Map。ConcurrentHashMap带来的结果是，在并发访问环境下将实现更⾼的吞吐量，⽽在单线程环境中只损失⾮常⼩的性能。

ConcurrentHashMap与其他并发容器⼀起增强了同步容器类：它们提供的迭代器不会抛出ConcurrentModificationException，因此不需要在迭代过程中对容器加锁。ConcurrentHashMap返回的迭代器具有弱⼀致性

（Weakly Consistent），⽽并⾮“及时失败”。弱⼀致性的迭代器可以容忍并发的修改，当创建迭代器时会遍历已有的元素，并可以（但是不保证）在迭代器被构造后将修改操作反映给容器。

尽管有这些改进，但仍然有⼀些需要权衡的因素。对于⼀些需要在整个Map上进⾏计算的⽅法，例如size和isEmpty，这些⽅法的语义被略微减弱了以反映容器的并发特性。由于size返回的结果在计算时可能已经过期了，它实际上只是⼀个估计值，因此允许size返回⼀个近似值⽽不是⼀个精确值。虽然这看上去有些令⼈不安，但事实上size和isEmpty这样的⽅法在并发环境下的⽤处很⼩，因为它们的返回值总在不断变化。因此，这些操作的需求被弱化了，以换取对其他更重要操作的性能优化，包括get、put、containsKey和remove等。

在ConcurrentHashMap中没有实现对Map加锁以提供独占访问。在Hashtable和synchronized-Map中，获得Map的锁能防⽌其他线程访问这个Map。在⼀些不常⻅的情况中需要这种功能，例如通过原⼦⽅式添加⼀些映射，或者对Map迭代若⼲次并在此期间保持元素顺序相同。然⽽，总体来说这种权衡还是合理的，因为并发容器的内容会持续变化。

与Hashtable和synchronizedMap相⽐，ConcurrentHashMap有着更多的优势以及更少的劣势，因此在⼤多数情况下，⽤ConcurrentHashMap来代替同步Map能进⼀步提⾼代码的可伸缩性。只有当应⽤程序需要加锁Map以进⾏独占访问 [1] 时，才应该放弃使⽤ConcurrentHashMap。

[1] 或者需要依赖于同步Map带来的⼀些其他作⽤。
