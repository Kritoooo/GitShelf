# 3.1 可⻅性

可⻅性是⼀种复杂的属性，因为可⻅性中的错误总是会违背我们的直觉。在单线程环境中，如果向某个变量先写⼊值，然后在没有其他写⼊操作的情况下读取这个变量，那么总能得到相同的值。这看起来很⾃然。然⽽，当读操作和写操作在不同的线程中执⾏时，情况却并⾮如此，这听起来或许有些难以接受。通常，我们⽆法确保执⾏读操作的线程能适时地看到其他线程写⼊的值，有时甚⾄是根本不可能的事情。为了确保多个线程之间对内存写⼊操作的可⻅性，必须使⽤同步机制。

在程序清单3-1中的NoVisibility说明了当多个线程在没有同步的情况下共享数据时出现的错误。在代码中，主线程和读线程都将访问共享变量ready和number。主线程启动读线程，然后将number设为42，并将ready设为true。读线程⼀直循环直到发现ready的值变为true，然后输出number的值。虽然NoVisibility看起来会输出42，但事实上很可能输出0，或者根本⽆法终⽌。这是因为在代码中没有使⽤⾜够的同步机制，因此⽆法保证主线程写⼊的ready值和number值对于读线程来说是可⻅的。

NoVisibility可能会持续循环下去，因为读线程可能永远都看不到ready的值。⼀种更奇怪的现象是，NoVisibility可能会输出0，因为读线程可能看到了写⼊ready的值，但却没有看到之后写⼊number的值，这种现象被称为“重排序（Reordering）”。只要在某个线程中⽆法检测到重排序情况（即使在其他线程中可以很明显地看到该线程中的重排序），那么就⽆法确保线程中的操作将按照程序中指定的顺序来执⾏。 [1] 当主线程⾸先写⼊number，然后在没有同步的情况下写⼊ready，那么读线程看到的顺序可能与写⼊的顺序完全相反。

在没有同步的情况下，编译器、处理器以及运⾏时等都可能对操作的执⾏顺序进⾏⼀些意想不到的调整。在缺乏⾜够同步的多线程程序中，要想对内存操作的执⾏顺序进⾏判断，⼏乎⽆法得出正确的结论。

NoVisibility是⼀个简单的并发程序，只包含两个线程和两个共享变量，但即便如此，在判断程序的执⾏结果以及是否会结束时仍然很容易得出错误结论。要对那些缺乏⾜够同步的并发程序的执⾏情况进⾏推断是⼗分困难的。

这听起来有点恐怖，但实际情况也确实如此。幸运的是，有⼀种简单的⽅法能避免这些复杂的问题：只要有数据在多个线程之间共享，就使⽤正确的同步。
